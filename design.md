# Design

## Overview

The project is split into two components:

- **`calendar_client_api`** — Abstract interface defining the contract for calendar operations.
- **`google_calendar_client_impl`** — Concrete implementation backed by the Google Calendar API.

## Why Two Components

The interface package has zero dependencies on Google libraries which mean that it will never depend on a specific provider. Swapping to a different calendar provider only requires a new implementation package.

## Dependency Injection

The DI pattern uses a simple registry in `calendar_client_api`:

- `register_client(factory)` stores a `Callable[[], CalendarClient]`.
- `get_client()` invokes and returns a new instance.

The implementation package auto registers itself at import time. When `google_calendar_client_impl` is imported, its `__init__.py` calls `register_client(get_google_calendar_client)`. After that, any call to `get_client()` returns a `GoogleCalendarClient`.

## Authentication

`GoogleCalendarClient` authenticates through a three-step chain, stopping at the first success:

1. **Environment variables** — `GOOGLE_CALENDAR_CLIENT_ID`, `GOOGLE_CALENDAR_CLIENT_SECRET`, `GOOGLE_CALENDAR_REFRESH_TOKEN`.
2. **Token file** — A previously saved `token.json` with OAuth credentials.
3. **Interactive OAuth** — Browser-based login using `credentials.json` (only when `interactive=True`).

No credentials are hardcoded. All secrets come from environment variables or files excluded from version control.

## Event Model

- `Event` (ABC) — Read only properties (`id`, `title`, `start_time`, `end_time`, `description`, `location`, `attendees`, `attachments`).
- `EventCreate` (dataclass) — DTO for creating events with all required fields.
- `EventUpdate` (dataclass) — DTO for partial updates.
- `GoogleCalendarEvent` — Concrete implementation that parses Google Calendar API payloads into the `Event` interface.

## Architecture Overview

**`google_calendar_service` (FastAPI Service):** A FastAPI application that wraps the existing `GoogleCalendarClient` and exposes it over HTTP endpoints. It handles OAuth authentication via browserbased login, manages user sessions with cookies, and delegates all calendar logic to the original implementation.

**`google_calendar_service_api_client` (Auto-Generated Client):** A Python client library autogenerated from the FastAPI service's OpenAPI specification using `openapi-python-client`. It provides typed Python functions that handle HTTP request construction, JSON serialization/deserialization, and response parsing. This eliminates the need for consumers to manually craft HTTP requests.

**`google_calendar_service_adapter` (Adapter):** An adapter that implements `CalendarClient` abstract interface by delegating to the autogenerated client under the hood. This is the key to transparency, consumer code that used `get_client()` continues to work identically, with no awareness that the implementation has moved from a local library to a remote service.

### Request Flow:

1. **Consumer** calls `client.create_event(EventCreate(title="Meeting", ...))`.
2. **Adapter (`ServiceCalendarClient.create_event`)** converts the `EventCreate` dataclass into an `EventCreateRequest`, mapping `Attendee` objects to `AttendeeRequest` objects and translating `None` optional fields to the generated client's `UNSET` sentinel.
3. **Generated client (`create_event_events_post.sync`)** serializes the `EventCreateRequest` to JSON, sends an `HTTP POST` to `/events`, and parses the JSON response into an `EventEnvelope` object.
4. **FastAPI service (`create_event` endpoint)** receives the request, converts the `EventCreateRequest` body back into an `EventCreate` dataclass via `event.to_event_create()`, and calls `client.create_event(event_create)` on the injected `GoogleCalendarClient`.
5. **`GoogleCalendarClient.create_event`** serializes the event to a Google Calendar API payload and calls `events().insert(...).execute()`.
6. **Google Calendar API** creates the event and returns the payload.
7. The response propagates back: Google payload -> `GoogleCalendarEvent` -> `EventResponse` (via `to_event_response`) -> `EventEnvelope` JSON → generated client's `EventEnvelope` object -> adapter unwraps to `ServiceCalendarEvent` -> returned to consumer as an `Event`.

## API Design

### Endpoints

| Method | Path | Description | Request Body | Response |
|--------|------|-------------|-------------|----------|
| `GET` | `/health` | Service health check | — | `StatusResponse` |
| `GET` | `/auth/login` | Redirect to Google OAuth consent screen | — | `302 Redirect` |
| `GET` | `/auth/callback` | Handle OAuth callback, exchange code for tokens | Query: `code`, `state`, `error` | `StatusResponse` |
| `POST` | `/auth/logout` | Clear session and OAuth tokens | — | `StatusResponse` |
| `GET` | `/events` | List calendar events | Query: `max_results` (default 10, min 1) | `EventsEnvelope` |
| `GET` | `/events/between` | List events in a datetime range | Query: `start`, `end` (ISO 8601) | `EventsEnvelope` |
| `GET` | `/events/{event_id}` | Get a single event by ID | — | `EventEnvelope` |
| `POST` | `/events` | Create a calendar event | `EventCreateRequest` | `EventEnvelope` |
| `PATCH` | `/events/{event_id}` | Partially update a calendar event | `EventUpdateRequest` | `EventEnvelope` |
| `DELETE` | `/events/{event_id}` | Delete a calendar event | — | `StatusResponse` |

### Request/Response Models

**`EventCreateRequest`** — Title, start_time, end_time, attendees, and attachments are required. Description and location are optional.

**`EventUpdateRequest`** — All fields are optional. Only provided fields are patched, omitted fields remain unchanged. This mirrors the `UNSET` sentinel pattern from `EventUpdate` in the API contract.

**`EventEnvelope`** / **`EventsEnvelope`** — Wrapper objects containing a single `EventResponse` or a list of `EventResponse` objects. This envelope pattern keeps responses extensible.

### Error Handling

Errors from the Google Calendar API and internal validation are translated into standard HTTP responses:

| Error Source | HTTP Status | Example |
|-------------|-------------|---------|
| Missing/expired OAuth session | `401 Unauthorized` | No valid OAuth tokens in session |
| Invalid OAuth state or missing code | `400 Bad Request` | Missing authorization code, invalid state |
| OAuth provider error | `400 Bad Request` | Provider returned `access_denied` |
| Pydantic validation failure | `422 Unprocessable Entity` | Invalid field types in request body |
| Token exchange network failure | `502 Bad Gateway` | Could not reach token endpoint |
| Token exchange provider error | `502 Bad Gateway` | Provider returned error or non-JSON response |
| Missing OAuth configuration | `500 Internal Server Error` | Missing `GOOGLE_CALENDAR_CLIENT_ID` env var |
| No session cookie | `403 Forbidden` | Session cookie not present or invalid |

## The Adapter Pattern

### Why It's Needed

The auto-generated client does not match the `CalendarClient` interface because:

1. **Method signatures differ.** The generated client uses module-level functions like `create_event_events_post.sync(client=..., body=...)` instead of instance methods like `client.create_event(event_create)`.

2. **Data types don't align.** The generated client uses its own `EventCreateRequest`, `AttendeeRequest`, and `EventResponse` models, while the `CalendarClient` interface expects `EventCreate`, `Attendee`, and `Event`.

4. **Response shapes differ.** The generated client returns `EventEnvelope`/`EventsEnvelope` wrappers, while the `CalendarClient` interface expects bare `Event` objects or iterables of `Event`.

### How It Works

The adapter consists of two classes:

**`ServiceCalendarClient`** implements `CalendarClient` and translates each method call:

```python
# Library usage (HW1) — direct import, runs in-process
import google_calendar_client_impl
from calendar_client_api import get_client, EventCreate

client = get_client()  # returns GoogleCalendarClient
event = client.create_event(EventCreate(title="Meeting", ...))
print(event.title)  # "Meeting"
```

```python
# Service usage through adapter, identical consumer code
import google_calendar_service_adapter
from calendar_client_api import get_client, EventCreate

client = get_client()  # returns ServiceCalendarClient (adapter)
event = client.create_event(EventCreate(title="Meeting", ...))
print(event.title)  # "Meeting" — same interface, same result
```

The consumer code is identical. The only difference is which package is imported at startup, `google_calendar_client_impl` HW1 library vs. `google_calendar_service_adapter` HW2 service. Both packages autoregister themselves with the DI registry on import, so `get_client()` returns the appropriate implementation.

Under the hood, the adapter's `create_event` method does the translation work:

```python
def create_event(self, event_create: EventCreate) -> ServiceCalendarEvent:
    # 1. Convert API types to generated client types
    attendees = [
        AttendeeRequest(email=a.email, name=a.name if a.name is not None else GEN_UNSET)
        for a in event_create.attendees
    ]
    body = EventCreateRequest(
        title=event_create.title,
        start_time=event_create.start_time,
        end_time=event_create.end_time,
        attendees=attendees,
        attachments=event_create.attachments,
        description=event_create.description if event_create.description is not None else GEN_UNSET,
        location=event_create.location if event_create.location is not None else GEN_UNSET,
    )
    # 2. Call generated client function (makes HTTP POST /events)
    response = create_event_events_post.sync(client=self._client, body=body)
    # 3. Unwrap envelope and return as Event-compatible object
    return self._unwrap_event_envelope(response)
```

**`ServiceCalendarEvent`** implements the abstract `Event` class by wrapping the generated `EventResponse`, handling the `Unset` -> `None` conversion for optional fields like `description` and `location`:

```python
@property
def description(self) -> str | None:
    if isinstance(self._event.description, Unset):
        return None
    return self._event.description
```

## Testing Strategy

### What We Tested

Testing is organized across the three components, plus crosscomponent integration and end-to-end tests:

| Component | What's Tested | Why |
|-----------|--------------|-----|
| `google_calendar_service` | FastAPI endpoints, OAuth flow, session management, settings validation | Verifies the service correctly wraps the client, handles auth, and returns proper HTTP responses |
| `google_calendar_service_adapter` | Type translation, UNSET mapping, envelope unwrapping | Verifies the adapter correctly bridges the generated client types to the CalendarClient interface |
| `calendar_client_api` (HW1) | ABC contracts, registry, event DTOs | Ensures the interface contract hasn't changed |
| `google_calendar_client_impl` (HW1) | Auth flows, CRUD operations, event parsing | Ensures the original implementation still works correctly |
| Integration tests | DI wiring, auto-registration, type hierarchies | Verifies components wire together correctly at runtime |
| E2E tests | Full CRUD lifecycle | Validates the entire system end-to-end |

### Test Types

The project uses three tiers of tests, enforced by pytest markers and run in sequence in CI:

**Unit tests** (`components/*/tests/`) — Fast, isolated tests per component. External dependencies are fully mocked. These verify individual component behavior in isolation.

**Integration tests** (`tests/integration/`) — Test DI wiring and cross-component interactions. Verify that importing `google_calendar_client_impl` auto-registers the factory, that `get_client()` returns the correct type, and that the type hierarchy (`GoogleCalendarClient`, `CalendarClient`) holds.

**E2E tests** (`tests/e2e/`):
- `test_application.py`: Runs against the real Google Calendar API using local credentials (marked `local_credentials`, skipped in CI).
- `test_application_ci.py`: Runs the full CRUD lifecycle against a mocked Google service resource, verifying the complete wiring without real credentials.

### Mocking Strategy

**What we mocked and why:**

- **Google Calendar API service resource:** Mocked with `MagicMock` in unit tests (`test_client_crud.py`) and CI E2E tests (`test_application_ci.py`). The Google API is external, ratelimited, and requires credentials, mocking it isolates our serialization/deserialization logic.

- **Authentication methods:** Each auth path (`_auth_from_env`, `_auth_from_file`, `_auth_from_interactive`) is individually mocked in `test_authentication.py` to test the auth chain without real credentials.

- **FastAPI `get_client` dependency:** Overridden with `app.dependency_overrides` in `test_main.py` to inject a `FakeCalendarClient`, isolating endpoint logic from auth and Google API concerns.

- **OAuth token exchange (`httpx.post`):** Mocked in `test_oauth.py` to simulate success, network errors, HTTP errors, and malformed responses without calling Google's token endpoint.

- **Session store functions:** Mocked in auth endpoint tests to verify session lifecycle (create, set handshake, consume, set tokens) without depending on the inmemory backend.

**What we tested with real implementations:**

- **Session store data model:** `SessionData` methods (`with_oauth_handshake`, `get_oauth_tokens`) are tested with real instances in `test_session_store.py`.
- **Event parsing:** `GoogleCalendarEvent` is tested with real Google-style JSON payloads to verify field extraction, timezone handling, and all-day event parsing.
- **Full CRUD lifecycle against live API:** `test_application.py` (when local credentials are available) runs create -> get -> list -> update -> delete against the real Google Calendar API.

### Interface Compliance

The adapter's compliance with the `CalendarClient` interface is verified through two approaches:

**Structural compliance (type system):** `ServiceCalendarClient` explicitly extends `CalendarClient` (the ABC), and `ServiceCalendarEvent` extends `Event`. Python enforces that all abstract methods and properties are implemented, the class cannot be instantiated otherwise. `mypy` in strict mode statically verifies that return types, parameter types, and property types match the abstract declarations.

**Behavioral compliance (tests):** The service endpoint tests in `test_main.py` use a `FakeCalendarClient` that implements the same interface and verifies that the FastAPI endpoints correctly serialize/deserialize between HTTP and the interface types. The integration tests in `test_client_integration.py` verify that `get_client()` returns an instance that satisfies `isinstance(client, CalendarClient)`. The CI E2E tests (`test_application_ci.py`) exercise the full CRUD surface through the `GoogleCalendarClient` with mocked backends, confirming that every method in the contract produces valid results. The DI integration test `test_di.py` explicitly registers a `GoogleCalendarClient` instance via the factory and asserts it satisfies the `CalendarClient` ABC.
